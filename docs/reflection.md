# Final Reflection
## Name: Althea Barbato

At the start of this course I had done some basic Linux stuff but had never managed a real server from scratch or written infrastructure as code. I really didn't know what Ansible was, I had never touched Prometheus or Grafana before, and I never really thought about what it actually takes to run something securely on the internet from start to finish. The idea of writing a deploy script and having it configure a server automatically felt like something that only happened at real companies with real teams. By the end of this course I have a publicly accessible server with a real Let's Encrypt TLS certificate, Prometheus monitoring with five alert rules, three Grafana dashboards, Uptime Kuma availability monitoring, automated nightly backups with a tested restore procedure, and full security hardening, all deployed with a single command from my laptop. That's a lot different from where I started at the beginning.

The biggest mindset shift across this course was learning to think about idempotency. In the beginning I was writing shell commands and scripts that would just break or behave weirdly if you ran them twice. Installing a package twice might give you an error. Writing a config file twice would override changes. Learning to write Ansible tasks that check current state before making changes meant that bash deploy.sh became something I could run over and over without worrying about what it would break. By the end of this project, running the deploy a second time came back all ok, changed=0. That result more than anything else made me feel like I actually understood what I was building rather than just running commands and hoping they worked.

Getting the monitoring stack set up in Lab 3 fundamentally changed how I thought about running a server. Before that lab, if something broke you found out when you tried to use it. After it, there are five Prometheus alert rules evaluating every 15 seconds and Uptime Kuma pinging the server every 60 seconds. When I stopped a Docker container to test the InstanceDown alert it fired within 30 seconds and showed up in Grafana. That made it feel real in a way that reading about monitoring never did. The server was actually watching itself like it was supposed to.

The backup and restore cycle was something I seriously underestimated. Writing the backup script was the easy part, maybe about an hour. The actual work was testing the restore. I moved the nginx config out of place on purpose, ran the restore script, and watched the site come back from backup. That test is what makes the backup story really credible. Anyone can write a script that creates a tar file. Knowing the restore works is a different thing entirely, and it only takes a few minutes to test. I should have done that kind of testing earlier in the labs.

The verify script pattern was something I came to appreciate more and more as the course went on. Early in Lab 1 and Lab 2 I was debugging by SSH-ing into the server and manually checking things one by one. By the final project, running bash scripts/verify.sh gives me 15 checks in about 30 seconds. Every section, SSH hardening, TLS, services, backups, kernel settings, all confirmed in one pass. That's a habit I'll definitely be keeping for anything I build in the future.

The three-layer firewall situation on Oracle Cloud caught me off guard multiple times and cost me a lot of hours across all the labs. I would open a port in UFW, try to connect, and get nothing back. Then I'd open it in iptables, still nothing. Then I'd finally remember that the Oracle VCN Security List is a completely separate firewall at the cloud level that I had to configure in the Oracle web console myself. All three layers have to agree for traffic to reach the server. Once I understood the architecture it made sense and I actually appreciated having that extra gate, but every time I started a new lab and forgot about the VCN I would lose time to this.

fail2ban is doing real work on this server and that wasn't obvious to me before running it on an actual internet-facing machine. I expected it to be a theoretical security control that might fire once in a while. The reality is that auth.log fills up with failed SSH login attempts from automated scanners within hours of any server going live on a public IP. fail2ban banning those IPs automatically means the noise stops and the logs become readable again. It is not a theoretical control. You can run sudo grep "Ban" /var/log/fail2ban.log and watch real IPs getting banned in real time.

auditd taught me to think carefully about what actually matters to watch rather than just logging everything. You could configure auditd to watch every file write on the system but then the logs are useless because they're full of normal system activity that completely overpopulates everything. Picking the four files that matter most for security, /etc/passwd, /etc/shadow, /etc/sudoers, and /etc/ssh/sshd_config, means that if something shows up in the audit log it's actually worth investigating. Focused logging is more useful than exhaustive logging.

The Let's Encrypt upgrade in the final project was more meaningful than I expected. The self-signed certificate from Lab 4 technically encrypts traffic but browsers display a security warning that makes the site look broken or dangerous. Anyone visiting would see a red warning before reaching the page. Getting a real certificate through certbot using the DuckDNS domain meant https://althea-lab.duckdns.org works in a browser exactly like any normal website, green padlock and no warnings. The auto-renewal cron job certbot installs means I don't have to remember to renew it. It just happens every 90 days automatically.

The custom Prometheus metric for SSH failed logins was the hardest debugging session in the whole course for me. I wrote a bash script that counted lines from auth.log and wrote a .prom file for node_exporter to pick up through the textfile collector. It looked correct to me, but Prometheus was returning a scrape error. After two hours I figured out that grep exits with return code 1 when it finds zero matches, which was triggering a fallback that appended an extra zero on a new line, making the .prom file have two values for the same metric, which is invalid Prometheus format. Switching from grep -c to wc -l fixed it because wc always exits successfully. That kind of bug is exactly what you only find by running things on a real server with real data.

The single server decision was the most obvious tradeoff for me. Everything running on one box means a single point of failure. If nginx crashes the site goes down. If the instance has a hardware issue, everything is gone. Oracle Cloud free tier actually gives two instances so setting up a load balancer with a second server is possible, but it would have doubled the Ansible role complexity and required setting up health checks, session handling, and coordination between two servers. For the scope of this project, accepting the single point of failure was the right call.

Docker with host networking was a deliberate simplicity choice that I'd probably revisit in a real production setup. The correct approach is Docker bridge networks with internal DNS so containers can only reach each other through defined paths. Host networking means every container can reach every port on localhost, which is a weaker isolation model. But bridge networking requires more configuration, internal service discovery, and debugging tools to understand what's happening. For a single-server monitoring stack in a course project, host networking is simple and it works. I knew the tradeoff and made the choice deliberately.

On-server backups are better than no backups but they have a fundamental problem: the backup and the data are on the same disk. If the disk fails catastrophically or Oracle terminates the instance, both the server data and the backups are gone. The right solution is to copy backups somewhere off-server after each run, an S3 bucket being the standard answer. That costs a small amount of money and adds complexity to the backup script, which is why I didn't include it. It's a documented known gap in the production readiness review, not something I missed. I just didn't want to pay for it.

Prometheus having no authentication is the infrastructure decision I'm least satisfied with. The /metrics endpoint is open to anyone who knows the server IP and port. The data is read-only system performance metrics so the actual harm is very low, but it's not good practice to have unauthenticated endpoints on a production server. The correct fix is to run nginx as a reverse proxy in front of Prometheus with HTTP basic auth. The reason I didn't do it is that the nginx configuration for proxying Prometheus without breaking Grafana's direct connection to it is more involved than it looks, and I ran out of time to get it right. It's an accepted risk with a known fix, I just didn't have time to cut it properly.

The most impactful future improvement would be off-server backups. A simple addition to the backup script that syncs the backup directory to an S3 bucket after each run would make the disaster recovery story complete. Right now, losing the server means losing the backups. With S3, losing the server means running the Ansible playbook on a new instance and restoring from the S3 bucket. That's real resilience.

After that, Prometheus authentication. Putting nginx in front of Prometheus with basic auth and updating the Grafana data source to include credentials would close the only major open security gap I have. It's about two hours of work once you understand the nginx proxy configuration.

A second server with a load balancer would eliminate the single point of failure. Oracle Cloud free tier provides two always-free compute instances so the compute cost is zero. The work is in updating the Ansible inventory and playbooks to manage two hosts, setting up health checks, and deciding how to handle the monitoring stack, which probably stays on one server.

Setting up DuckDNS and Let's Encrypt from Lab 1 instead of switching at the end would have been much cleaner. The cert migration worked fine but having a real domain from the beginning would mean every lab had a properly trusted HTTPS setup from the start.

The Ansible structure I ended up with is much closer to how real infrastructure teams work than I expected when I started the course. Roles that each handle one concern, handlers that only trigger on actual changes, idempotent tasks, shared variable files, separate playbooks for logical groups of work. The pattern of writing configuration locally, deploying to the server, and running automated checks to verify the deployment is basically what CI/CD pipelines do, just without the automated triggering. Understanding that pattern at this level makes the jump to actual CI/CD much smaller.

Prometheus and Grafana are real production tools used by actual companies. The monitoring stack I set up is the same stack that powers observability at organizations of all sizes. The dashboards I built are basic but the concepts are the same: scrape targets, metrics, PromQL queries, alert rules, notification channels. Understanding how they fit together, how scraping works, what a textfile collector is for, how Grafana provisioning as code works, is a real foundation that applies directly to professional infrastructure work.

The security hardening process gave me a practical checklist that applies to any Linux server. Harden SSH, install fail2ban, install auditd and configure meaningful watch rules, apply sysctl kernel hardening, remove unnecessary services and packages, enable automatic security updates. That list is not specific to this project or this class. Every Linux server I set up going forward starts from that checklist.

This was one of the most genuinely hands-on technical things I've done in a school class. There's a significant gap between reading about SSH hardening and actually configuring it on a real server with automated scanners hitting port 22 within hours. Reading about monitoring and watching a Prometheus alert actually fire when you stop a container. Reading about backups and breaking your own nginx config on purpose to test whether the restore actually works. The course structure of building incrementally across four labs meant that each piece had context and its own purpose. By the final project I wasn't configuring things because the assignment said to. I was configuring them because I understood what they were protecting against. And that understanding is what I'm keeping from this course.
